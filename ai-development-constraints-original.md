# AI 开发通用约束

## 1. 适用范围

本文件用于约束 AI 在软件项目中的需求分析、设计、编码、测试、调试、依赖管理、文档更新、发布准备和问题回收等行为。  
适用于新功能、缺陷修复、重构、配置变更、接口调整、数据库迁移、脚手架生成和线上问题处理。

## 2. 总原则

- 先理解，再修改。
- 先证据，再结论。
- 先最小变更，再扩大范围。
- 先更新契约，再改实现。
- 先验证，再宣称完成。
- 先脱敏，再落盘。
- 先说明偏离，再执行偏离。

## 3. 需求与范围约束

- 任何未确认的需求都只能作为假设，不能直接当作事实实现。
- 不得擅自扩大范围，例如自动增加支付、短信、复杂权限、多租户、消息队列、微服务拆分等。
- 每个新增能力都必须说明业务理由、依赖代价、验证方式和回退方案。
- 如果需求存在歧义，优先收敛到可验证的最小方案。
- 当设计发生变化时，必须同步说明哪些原始假设失效。

## 4. 设计与架构约束

- 设计先于实现，且必须形成可执行的约束或接口。
- 关键边界必须显式化：模块职责、输入输出、状态流转、失败处理、权限边界、数据流向。
- 不得在没有证据的情况下改架构、改目录、改协议或改运行策略。
- 任何架构偏离都必须写明原因、替代方案、风险和验证方法。
- 不得将临时绕过当作永久方案。

## 5. 文档约束

- 只要实现决策改变了依赖、命令、目录、协议、状态机、错误码、数据库、日志、测试或启动方式，就必须同步更新对应文档。
- 文档必须区分：
  - 事实
  - 假设
  - 约束
  - 风险
  - 验证结果
- 不得删除失败证据；失败日志、blocked 报告、测试输出和修复说明必须保留。
- 文档中的示例必须和真实实现一致，不得用过时样例误导后续开发。

## 6. 依赖与版本约束

- 禁止把精确版本写成 `latest`。
- 版本选择以稳定兼容线为准，不追逐最新主版本。
- 依赖升级或降级必须说明：
  - 原版本
  - 新版本
  - 触发原因
  - 兼容影响
  - 验证命令
- 不得通过无关依赖一起升级来掩盖根因。
- 不得只改 lockfile 不解释原因。
- 优先使用标准库或成熟第三方库，只有在证据充分时才自研。

## 7. 命令与环境约束

- 每条命令必须明确工作目录。
- 跨目录命令必须拆分，不能混成一条含糊 shell 字符串。
- 不得把用户输入直接拼接进 shell 命令。
- 默认使用非 shell 执行方式，只有明确需要时才允许 shell。
- 环境变量必须最小化传递，禁止无过滤透传整个进程环境。
- 本地缓存、虚拟环境和构建缓存优先放在项目可控路径内。

## 8. 协议与接口约束

- 先更新协议、schema、接口定义，再改实现。
- 机器判定依据必须是结构化数据，不依赖自然语言 stdout。
- 旧字段弃用必须给出兼容策略，不能无说明删除。
- 报告、状态文件、任务图和日志格式必须可校验、可回放、可追踪。
- 接口变更必须同步更新测试夹具和调用方。

## 9. 状态与流程约束

- 状态机必须显式定义合法流转。
- 任务图必须是 DAG，不能只是待办列表。
- 失败、重试、返工、blocked 都必须有明确阈值和处理路径。
- 不允许把失败状态直接带入后续阶段。
- 需要人工确认的阶段要明确标出，不能让 AI 自行跳过。

## 10. 代码实现约束

- 小步实现，避免一次改动覆盖过多模块。
- 只改必要文件，不做无关重构。
- 注释只解释意图、约束、风险和非显然设计，不复述代码表面行为。
- 正则、并发、权限、配置和 workaround 必须写清原因。
- 删除、重命名或迁移接口时必须考虑兼容层。

## 11. 测试与验收约束

- 验证顺序优先为：安装、lint、单测、集成测试、构建、E2E、启动、验收。
- 测试失败必须给出失败命令、退出码、日志路径和修复点。
- 不得通过删测、skip 测试或伪造结果来掩盖问题。
- 通过不等于完成，必须有对应产物、报告和证据落盘。
- 生产前的关键门禁必须可重复执行。

## 12. 安全约束

- 不得写入真实密钥、密码、token、Cookie、Authorization header。
- 日志、报错、报告和示例都必须先脱敏。
- 环境变量要白名单化，不能把整份环境直接传给命令或 Agent。
- API 暴露必须分级管理：公网、内网、管理端、调试端、健康检查、文档页分别定义。
- 鉴权、授权、CORS、CSRF、限流、IP 白名单、反向代理和审计日志必须在设计阶段明确。
- 生产环境不应默认公开调试接口、内部管理接口或完整接口文档。
- 出现真实凭据需求时，应进入明确人工处理点，而不是要求在日志里粘贴。

## 13. 数据与数据库约束

- 数据库迁移、schema、seed 和容器版本变更必须同步说明。
- 破坏性 schema 变更必须有风险评估和回退策略。
- 不得默认切换数据库引擎或 ORM。
- 数据库字段、日志字段和接口字段涉及敏感信息时，必须统一脱敏与留存策略。
- 删除数据、修复数据、导出数据都要有明确权限和审计要求。

## 14. 发布与上线约束

- 每次发布都必须有版本、变更摘要、验证结果和回滚思路。
- 高风险变更应支持开关、灰度或分阶段发布。
- 上线后要关注健康检查、错误率、慢请求、权限拒绝和外部依赖失败。
- 没有验收证据，不得宣称可上线。

## 15. AI 行为约束

- 不得伪造测试结果、运行结果或验证命令。
- 不得删除失败证据来制造成功假象。
- 不得把猜测当成事实。
- 不得在未理解代码前直接大改。
- 不得用大范围重构掩盖局部问题。
- 不得在缺少证据时声称“已完成”“已验证”“已修复”。

## 16. 最终交付约束

每次完成开发任务，最终说明至少应包含：

- 改了哪些文件
- 为什么改
- 是否偏离原文档
- 如果偏离，更新了哪些文档
- 运行了哪些验证命令
- 仍然存在的风险

## 17. 推荐的补充清单

如果要把本文件作为项目级强制规范，建议继续补上这些专题：

- 生产安全规范
- 权限矩阵规范
- 发布回滚规范
- 数据保留与删除规范
- API 兼容性规范
- 观测与审计规范
- AI 交付报告模板
